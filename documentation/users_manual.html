How to run ParaGaus





The ttfs Script

The ttfs script is the driver script of the ParaGauss Program. Calculations can only be started using this script.

usage 

 ttfs_version \
   name \
   [ -name_is_extension ] \
   [ -extend_outputnames ] \
   [ -inputdir inputdir ]  \
   [ -outputdir outputdir ]  \
   [ -dir dir ] \
   [ -newinput filename ]  \
   [ -gxfile filename ] \
   [ -restartable ] \
   [ -save_old_output ] \
   [ -zip ] \
   [ -vpp_mbx_size  number ] \
   [ -us |
     -pd ] \
   [ -shared_fs  dir ] \
   [ -recover_in_datadir ] \
   [ -keep_slaveoutput ] \
   [ -keep_core ]  \
   [ -profile |
     -count  sourcedir |
     -vpp_stats  optionnumber ]  \
   [ -vers version ] \
   [ -scriptvers version ] \
   [ -interfacevers version ] \
   [ -simolvers version ]  \
   [ -opt geometry optimization program ] \
   [ -integral |
     -integral_split |
     -split | 
     -split_scf |
     -scf |
     -interfacefile |
     -paragau |
     -help ]  \
   [ -datadir datadir ] \
   [ -rm_datadir |
     -keep_datadir ] \
   [ -oldinput filename ]  \
   [ -interfaces filename ]  \
   [ -lcgtooptions option [option ...] ]

For obtaining on-line help: 

 ttfs_version -help

The options listed above may appear in arbitrary order except for -lcgtooptions that must appear last. All options are optional. The first argument name as described below is obligatory.

The following options are SP2-specific:

  • [ -us | -pd ]
  • [ -shared_fs dir ]

The following options are VPP-specific:

  • [ -profile | -count sourcedir | -vpp_stats optionnumber ]
  • [ -vpp_mbx_size number ]

location and naming of input and output files

The ttfs script offers flexibility in naming and locating input and output files. Principally, two philosophies are supported for separating files of different calculations:

one specific directory for each calculation
use name to specify directory (absolute path recommended). No file name extensions are necessary.
several calculations in one directory kept apart by file name extensions.
use name to specify file name extensions and set -name_is_extension. Files are located either in present working directory or the one specified with -dir dir.
Separating input and output directory and naming files individually is also supported via the switches described below.

name
The meaning differs depending if you set -name_is_extension.
  • In case you do not set this flag, name is the default input and output directory of your calculation. Both relative and absolute paths are valid, but the absolute one is recommended. The last part of name is also a valid postfix for input files that allows to keep input files for different calculations in one input directory apart. In case you set -extend_outputnames it will also be the postfix to all output files.
  • In case you set this flag, name is a postfix to all input and output files. The directory where your input and output files are located is the present working directory unless you specify other with -dir or -inputdir and -outputdir.
Intermediate directories used during the calculation are also termed according to name.
[ -name_is_extension ]
name is not used to set inputdir and outputdir but only as extension to input and output filenames. This implies automatically -extend_outputnames. Note that input files without extensions are also accepted if none with extension exist.
[ -extend_outputnames ]
use last part of name as extension to output files.
[ -inputdir inputdir ]
directory that holds input files. Default is name, of if -name_is_extension is given present working directory.
[ -outputdir outputdir ]
directory where to store output files of new program run. If directory already exists old output files will be overwritten. Default is name, of if -name_is_extension is given present working directory.
[ -dir dir ]
equivalent to -inputdir dir -outputdir dir.
[ -newinput filename ]
input file in inputdir to new program. Default is file input.
[ -gxfile filename ]
gx file in inputdir containing geometry and Z-matrix for geometry optimization. Default is gx.

various useful options

[ -restartable ]
leave gx file and scf restart files in place instead of renaming them. This allows to restart calculation without modifying input/output directory. Default is moving these files to protect them from accidental modifications. The switch also implies -save_old_output (see below).
[ -save_old_output ]
move old output files (output, input.out, gx.start, trace file) to files with _number appended where number is the lowest one without existing files. All files of one run will obtain the same number.
[ -zip ]
compress output files
[ -vpp_mbx_size number ]
set size of mailbox for incoming PVM messages on VPP. number can take the following values resulting in the given mbx sizes on each host:
14 MB
416 MB
1664 MB (this is the default)
64256 MB
[ -shared_fs dir ]
use dir as shared scratch file system on SP2"
[ -recover_in_datadir ]
fetch and/or write scf recover files to datadir during calculation. Default is to store them in inputdir during calculation.

debug options

[ -us ]
use user space PVMe instead of Internet protocol PVMe on SP2"
[ -pd ]
use public domain PVM instead of Internet protocol PVMe on SP2"
[ -keep_slaveoutput ]
copy the slave output directories to the directoies termed after the hostnames or 1, 2, ... created in outputdir.
[ -keep_core ]
keep the core file for debugging in case new program crashes.
[ -profile ]
produce profiling information. Perform profiling by statistically recording position of program. Only available on VPP for 1 processor run.
[ -count sourcedir ]
produce profiling information. Count number of execution of loops and store vector lengths. Only available on VPP for 1 processor run. sourcedir must be directory containing the source files the executable was build from. Attention: an executable compiled with flag -Wc and linked with -lfjcnt is required.
[ -vpp_stats option_number ]
produce profiling information of parallel run. Only available on VPP. option_number can be a sum of the following flags:
1Global performance (CPU times, instructions count...)
2Communication overhead (total send/receive times,...)
4Global profile (Top routines, top ten per processor)
8Detailed profile per processor
16Extended profile (detailed plus system calls)
32Unix background activity
64IO traces
128No batch output (keep trace files for MPTools)

selecting versions of called programs

The versions of various programs called by the ttfs script can be selected individually. For normal purpose, the defaults should be fine. In most cases, the programs use the same version as the ttfs_version script.

[ -vers version ]
version of ParaGauss program to use Default is version of ttfs_version script.
[ -scriptvers version ]
version of help programs to use Default is version of ttfs_version script.
[ -interfacevers interfacevers ]
version of the program read_ftn1 to use Default is version of ttfs_version script.
[ -simolvers simolvers ]
Version of Simol-Program. Default is _v170. If a TS-Search is to be performed or a minimum has to be found using the eigenmode following, _v180 is recommended. If you want to calculate frequencies, please specify _v170ttfs.
[ -opt geometry optimization program ]
Can be either 'simol' or 'optimizer', the new optimization program. However, 'simol' is still strongly recommended. This is also the default.

previous execution of old lcgto

The use of the options described in this chapter is no longer necessary for normal purpose calculations except for the switch -help that gives an on-line help.

The previous execution of old lcgto is not possible on the VPP.

The ttfs script allows to convert inputs from old lcgto and to use precalculated integrals from old lcgto. If this is really necessary is determined automatically by reading the namelists TASKS or OPERATIONS from the input file and checking if the input file contains a line "read intermediate input" that makes the program read in an intermediate file produced by a special version of old lcgto.

In case a previous run of lcgto is found necessary the default is to execute this run and the ParaGauss run all at once. The script can also execute the process in several steps:

  1. For running only the scf part of new program with precalculated integrals, three steps are necessary:
    1. running old lcgto to calculate integrals
    2. splitting of integral files according to number of processors desired
    3. distribution of integral files to hosts and run of new program for SCF procedure
    Step 1 and 2 are not parallel, but step 3 is. It is necessary to specify the location where the integrals are kept with [ -datadir dir ].
  2. For converting input but running the integral part of new program, two steps are necessary:
    1. running old lcgto to calculate interface information
    2. setting up of directories and run of new program (only if input file requests interface file)
    Step 1 is not parallel, but step 2 is. Please note that ISTOP 2 in input to old lcgto is enough.
[ -integral | -integral_split | -split | -split_scf | -scf | -interfacefile | -paragau | -help ]
tasks as described above:
-integralA.1
-integral_splitA.1 + A.2
-splitA.2
-split_scfA.2 + A.3
-scfA.3
-interfacefileB.1
-paragauB2
-helpgives on-line help
Defaultcomplete run (A.1 + A.2 + A.3 / B.1 + B.2 / only ParaGauss)
[ -datadir datadir ]
directory that holds or will receive integral files and output from old lcgto This directory can be reused from previous runs and must be specified if no integrals are calculated. Look at end of output of previous run (TTFSDATADIR) to obtain this directory.
[ -rm_datadir | -keep_datadir ]
remove or keep datadir TTFSDATADIR that holds integrals when calculation is finished. Default is to keep this directory and giving its name in the end of calculation for mode A and removing the directory for mode B.
[ -oldinput filename ]
input file to old lcgto with ending .inp omitted in inputdir. Please note that for mode B ISTOP 2 in input to old lcgto is enough. Default is taking the file name.inp or the only file ending with .inp in directory if it does nor exist.
[ -interfaces filename ]
interfaces input to new program. If specified, interfaces file will be located in inputdir Default is file interfaces in directory datadir for mode A and inputdir for mode B
[ -lcgtooptions option [option ...] ]
Options to pass to old lcgto script. These options must be given last of all ! Default is v168_beta -exedir /users/hp4/ttfs/bin -kit

exit status

The ttfs script returns zero in case of successful execution and other values in case of errors. In the case of geometry optimization, zero is returned as long as the optimization did not yet converge within the number MAX_GEO_ITERATION of points allowed and 100 if the optimization converged.

This is useful in the case of geometry optimization. If not all the geometry optimization cycles can be performed at once due to time limits of the queue system, the exit status of the ttfs script can be used for automatic resubmission of the job.

The exit status of a program can be obtained via the shell variable "$status" directly after the execution of the program.

ttfs_version name

Running a simple calculation with version version of ttfs script. All input and output files are located in the separate directory name that is only used for this calculation. The file names have no postfix extension. Old output files will be overwritten.


ttfs_version name -name_is_extension -dir dir -zip -save_old_output -vers mainversion

Running a calculation with version version of ttfs script. All input and output files are located in directory dir that is used for several different calculations. The filenames have the postfix extension name that is used to keep the different calculations apart. Output files will automatically be compressed. Old output files of a previous run will be moved to filenames with _number appended with number being the lowest number not yet used. All files of one run obtain the same number. For instance, the previous output file "output.name.gz" would be moved to "output.name_number.gz". The version mainversion of the ParaGauss executable will be used.


ttfs_version name -restartable

Running a geometry optimization calculation with version version of ttfs script. All input and output files are located in directory name that is only used for this calculations. The filenames have no postfix extension. Old output files of a previous run will be moved to filenames with _number appended with number being the lowest number not yet used. All files of one run obtain the same number. For instance, the previous output file "output" would be moved to "output_number". Recover files from the last run are not moved and will be reused as a start for the SCF procedure (See RECOVER_OPTIONS). This is useful when not all the geometry optimization cycles can be performed at once due to time limits of the queue system. The exit status of the ttfs script could be used for automatic resubmission of the job.





Input and Output Files

See chapter "location and naming of input and output files" in the description of the ttfs script for the location of input and output files. "name" referenced below is the name of the calculation described there. Input files and intermediate files are allowed to have the postfix extension ".name". Output files, intermediate files, and run time control files will also obtain this extension if the flags -name_is_extension or -extend_outputnames were given as arguments to the ttfs script.

Input Files

input
ParaGaus input file as described below The name can be altered by argument -newinput to the ttfs script.
gx
gx file containing geometry and Z-matrix for geometry optimization. This file will be updated automatically by the geometry optimizer at the end of each geometry loop. The name can be altered by argument -gxfile to the ttfs script.
name.inp
Outdated. Input file to old lcgto if previous run of old lcgto is done (See chapter "previous execution of old lcgto" in ttfs script part). The name can be altered by argument -oldinput to the ttfs script.

Output Files

output
ParaGauss main output file.
input.out
This file contains an expanded form of the input. It is in the input format and can directly be used as input for a new run. It protocolls all the options used, also all options implied by defaults. This file can also be used as a starting point for editing. The amount of information written to the file is controlled by the switch ECHO_INPUT_LEVEL in the namelist TASKS.
trace_output
This file is used to monitor the progress of the calculation. However, at the end of the calculation it also contains an table giving an overview over the SCF-cycles. Moreover, in the case of geometry optimization the updated gx file will be appended to this file at the end of each geometry loop (not on the VPP).
debug_output
For debug output only. Only generated in case you set IOUNITADMIN_USE_DEBUG in namelist OUTPUT_IOUNITADMIN.
buggy_log
For debug output only. Only generated in case you set IOUNITADMIN_USE_BUGGY in namelist OUTPUT_IOUNITADMIN. This file is well suited for finding the exact location of a program crash since it is always opened and closed.
gx.start
When starting a geometry optimization, the original gx file "gx" is copied to "gx.start".
listing
The listing of the contents of the temporary directory used by the master during the calculation at the end of the calculation.
simol/trace_simol
In case of geometry optimization on the VPP, this file is used to append the output of the optimizer and the updated gx file at the end of each geometry loop.
simol/dfp.name
In case of geometry optimization, this file is used to append the output of the optimizer at the end of each geometry loop.
name.out
Outdated. Putput file of old lcgto if previous run of old lcgto is done (See chapter "previous execution of old lcgto" in ttfs script part).

Intermediate Files

These files will be located in the input directory in case input and output directories are separate.

During SCF Iterations, it is possible to save and recover the state of the calculation at various points of the SCF loop. This option is both useful for recovering crashed calculations and to save computing time by starting the SCF part with a good approximation of the orbitals when performing a geometry optimization. The files "saved_*.dat" listed below are used for this purpose. The namelist RECOVER_OPTIONS controls if and which files are written / read. The files will automatically be compressed and are binary.

saved_ksmatrix.dat
SCF-recover file as described above: hamiltonian
saved_eigenvec.dat
SCF-recover file as described above: eigenvectors
saved_scfstate.dat
SCF-recover file as described above: charge fit coefficients and exchange part of hamiltonian
saved_fitcoeff.dat
SCF-recover file as described above: fit coefficients
simol
In case of geometry optimization, this directory contains intermediate and output files of the simol program.
name.gsn name.fl9 name.fl10
In case of geometry optimization, these files are used by the simol program to store intermediate values.
interfaces
Outdated. Interface file produced by old lcgto and read in in case input file contains line "read intermediate input". (See chapter "previous execution of old lcgto" in ttfs script part). The name can be altered by argument -interfaces to the ttfs script.
eigenvectors.dat occ_num.dat
Outdated. The files "eigenvectors.dat" and "occ_num.dat" are produced by old lcgto. These file may not be compressed. They are only read if integrals are taken from old lcgto as well. (See chapter "previous execution of old lcgto" in ttfs script part).

Runtime control Files

These files will be located in the output directory in case input and output directories are separate.

trace_output
This file is used to monitor the progress of the calculation. It is always opened and closed and so always shows the actual task the program is executing. It also contains a table giving an overview over the SCF-cycles.
scfcontrol
This file only exist while the SCF part is executed. It is automatically generated and read in each SCF-cycle. It allows to change certain settings made in the input file at run time. It contains the namelists CONVERGENCE_LIST, MIXING, RECOVER_OPTIONS and FERMI described in the Input Description below. To abort a running calculation, edit the namelists CONVERGENCE_LIST: Set MAX_ITERATION below the present SCF-cycle number and set STOP_IF_NOT_CONVERGED.

Spin Orbit

Spin-Orbit (SO) calculations is another step further after SR calculations. Up to date it is the only possibility to do single-point SO-calculations.

There are several SO modes possible:

  • So-called "plain" spin-orbit where only the kinetic energy and nuclear attraction are corrected for relativistic effects. This is an SO counterpart of SR calculations.
  • One can turn relativistic transformations of coulomb (hartree) term on - so called Vee run. One can do that to different levels of accuracy:
    • first order DK transformation of hartree term
    • two flawors of the second order DK

To start an SO calculation one needs:

  • make sure that you are going to do a single-point calculation - check namelists OPERATIONS and TASKS
  • turn on RELATIVISTIC and SPIN_ORBIT switches in namelist MAIN_OPTIONS. Note that SO calculations are always 'spin-restricted' even if 'polarized', so make sure that switch SPIN_RESTRICTED is TRUE in this namelist.
  • in the namelist SPIN_ORBIT specify what level of SO you want.
  • you may want to speed up you calculation by setting XC="...,spatial" in XC_CONTROL
  • you may want to set XC = "vwn,gga:all" in PHXC_CONTROL namelist




Submitting Jobs

HP

On the HPs, ParaGauss must be run interactively. Only users assigned a special permission to run long jobs without being reniced by the superuser can do so.

Before starting ParaGauss, a PVM deamon must be running on all hosts. To start the daemon in case it is not yet running, perform the following steps:

  • Enter PVM console (This will start the daemon in case it is not yet running):
    > pvm
  • Check if all host are included in the virtual machine:
    pvm> conf
  • Add the missing hosts (use "hostname+atm" since we use a special version of PVM adapted to use our ATM network). For instance:
    pvm> add hp1atm
  • Leave the PVM console letting the deamon alive:
    pvm> quit

After that, you can directly start ParaGauss by calling the ttfs script from the command line or by executing a script file containing the call to the ttfs script.

After ParaGauss finished, you can kill the PVM deamon:
> pvm
pvm> halt

Geometry Optimization

gx file

This file contains geometry and Z-matrix for the calculated geometry. This file is used to communicate between the ParaGauss program and the geometry optimizer. ParaGauss takes its input geometry from this file. At the end of each geometry loop, ParaGauss appends the calculated energy and gradients to the file. Then it invokes the geometry optimizer that reads in the file, calculates a new geometry and updates the gx file. After that, ParaGauss reads in the file once more at the beginning of the next geometry loop.

Input Description





Format

The input format is intended to be as free as possible.

The following special features are supported:

Comments
Everything after character "#" in line is ignored
Empty lines
... are ignored.
Continuation lines
The preceding line must contain "%". Everything after this character is ignored and the next line is read. This is helpful when formatting lines containing numbers. Namelist may be written in several lines anyway.
Include Files
"~" at the beginning of line followed by filename makes this file to be read. When the end of the include file is reached reading continues in the previous file after the include line. Recursive includes are also allowed, i.e the include files may contain new include directives.
Masking special characters
"#", "%" and "\" can be masked with an "\" preceding it.
Number Formats for Real Numbers
There are no fixed formats. Any normal notation will be accepted. For instance:
  • 1.001
  • 1
  • 1.0E00-1


The main building blocks are so called "namelists" that obey the following rules:

  • Namelists start with the character "&" directly followed by the name of the namelist.
  • After that there are statements of the kind "Variable=Value". Where
    • Variable is the name of a variable that is included in the namelist.
    • Value is a value assigned to the variable. The format depends upon the type of the variable used:
      • For Numbers any notation as described above can be used
      • Strings must be quoted. They are case-sensitive.
      • Logicals can be assigned the following values: T F TRUE FALSE .T. .F. .TRUE. .FALSE.
    • An arbitrary number of blanks (including 0) is allowed before and after "="
  • Statements are separated by one or more of the following items:
    • One or more blanks
    • the character ","
    • A new line
  • The first statement and the Namelist name are separated by one or more of the following items:
    • One or more blanks
    • A new line
  • Namelists end with the character "/". Everything after that in the line is ignored. For clarity, it is good practice to repeat the name of the namelist here.
  • Namelists may be written in one or more lines.
  • Namelists are case-insensitive, i.e. using capital or minor letters makes no difference.
  • The order in which the statements appear is free.
  • It is not necessary to assign values to all variables in the namelist. Variables that are omitted will be assigned default values. However, certain variables specially denoted in the description of the namelists below do not have default values and need to be specified.

Here is an example: 

 &NAMELIST_NAME
    int_var=1, REAL_VAR=0.7E1
    String_Var = "Hello"
 /NAMELIST_NAME




Syntax

The input consists of namelists, sometimes directly followed by lines containing lists of numeric parameters (This is mentioned for each of the relevant namelist individually in the explication of namelists below.).

Most of the namelists can appear in arbitrary order, but some have to appear together in blocks with fixed order:

Molecular Geometry
The namelists UNIQUE_ATOM_NUMBER and UNIQUE_ATOM have to be grouped together as described with namelist UNIQUE_ATOM_NUMBER.
Point Charge Geometry
The namelists POINTCHARGE_NUMBER and POINTCHARGE have to be grouped together as described with namelist POINTCHARGE_NUMBER
Basis Set
The namelists UNIQUE_ATOM_BASISSET, UNIQUE_ATOM_BASIS and UNIQUE_ATOM_GLOB_CON have to be grouped together as described with namelist UNIQUE_ATOM_BASISSET.
SCF Grid
The namelists GRID and GRIDATOM have to be grouped together as described with namelist GRID.
Post Hoc Grid
The namelists GRID_PH and GRIDATOM_PH have to be grouped together as described with namelist GRID_PH. In case this optional block does not appear the settings from the SCF grid are used.
Output Options
The namelists OUTPUT, OUTPUT_CONFIG, OUTPUT_DIPOLE, OUTPUT_INTEGRAL, OUTPUT_IOUNITADMIN, OUTPUT_POST_HOC, OUTPUT_SCF, OUTPUT_SYM, OUTPUT_TIMING, OUTPUT_TRACE, and OUTPUT_UNIQUE_ATOMS have to be grouped together as described with namelist OUTPUT. In case it appears, the optional namelists OUTPUT must start the block. The remaining optional namelists may appear in arbitrary order.
Information on Irreps
This block is only required in the outdated case the new symmetry part is not used. The namelists SYMMETRY_DATA and IRREP have to be grouped together as described with namelist SYMMETRY_DATA.
Symmetry Adaption Coefficients
This block is only required in the outdated case the new symmetry part is not used. The namelists UNIQUE_ATOM_INDP_FCT and UNIQUE_ATOM_SYMADAPT have to be grouped together as described with namelist UNIQUE_ATOM_INDP_FCT. In case no integrals are calculated this block also contains renormalization coefficients.
The optional user conctrolled population analysis block
It consist of namelist POPULATION followd by possibly several namelists POCOLUMN as described with namelist POPULATION.
The optional fragment orbital analysis block
It consist of namelist FRAG_ORB_ANALYSIS followd by namelists FRAGMENT. as described with FRAG_ORB_ANALYSIS.
The optional orbital plot block
It consist of namelist ORBITAL_PLOT followd by namelists ORBITAL_LIST as described with ORBITAL_PLOT.

Some additional rules concerning the order of the remaining namelists and the blocks listed above have to be observed:

  • Either the namelist TASKS of the outdated namelist OPERATIONS must start the input. Those namelists are mutually exclusive.
  • The molecular geometry block contains the specification of the number of unique atoms. Thus it must precede the following blocks that need this information:
    • Basis Set
    • SCF Grid
    • Post Hoc Grid
    • Symmetry Adaption Coefficients
  • The outdated Symmetry Adaption Coefficients block must be preceded by the following blocks:
    • Molecular Geometry
    • Information on Irreps
    • Basis Set

Each of the blocks listed above and each of the remaining namelist not contained in a block may only appear once.

Most of the namelist and blocks are optional and default values are used if they are omitted. However the following namelist and blocks are required:

  • Either the namelist TASKS of the outdated namelist OPERATIONS must start the input. Those namelists are mutually exclusive.
  • The Molecular Geometry block.
  • The Basis Set block.
  • The namelist PVMOPTIONS to specify the number of processors.
  • The use of the optional SCF Grid block and the optional MAIN_OPTIONS is highly recommended.
  • In the outdated case no symmetry part is run the blocks Information on Irreps and Symmetry Adaption Coefficients are also required.

The outdated use of the Information on Irreps block and the Symmetry Adaption Coefficients blocks is only allowed in case the symmetry part is not run. (See namelist OPERATIONS.) In this case the namelists SYMMETRY_GROUP is forbidden.

It is possible to convert old lcgto inputs with the help of the ttfs script. In this case an intermediate file will be generated by a special version of lcgto. To read in this intermediate file the input file must contain one line "read intermediate input". The Molecular Geometry block, the Basis Set block, the Information on Irreps block and the Symmetry Adaption Coefficients block or the namelists SYMMETRY_GROUP must all be omitted since this information will be taken from the intermediate file. Note that other information from the old lcgto input is not converted (for instance: grid, exchange potential ...) and must be specified manually in the input file.

In case of errors in the input file, an error message giving the filename and the line number of the last processed line and showing this line will be produced when the first error is detected and the program will stop. Consecuting errors are not detected.

DIPOLE_TRANSITIONMOMENTS

This namelist selects which dipole transition moments should be printed.

There is also the possibility to create intensities for Xray Emission Spectra (XES). The Xray emission is caused by the recombination of a valence orbital and a previously generated core hole. In the theoretical approach the calculation is performend in the frozen orbital groundstate approximation. The data will be written to the file xes.dat in the output directory and will contain 2 columns. The columns are the Kohn Sham eigenvalue (in au) of the valence orbital and the intensity. The intensity is calculated, according to Fermi's Golden rule as the product of the square of the dipole transition matrix element and the difference between the core orbital and the valence orbital to the power 4..

This namelist only has effects if dipole calculation are switched on in namelist TASKS or OPERATIONS or PROPERTIES and OUTPUT_DIPOLE_TRANSITIONM_F is switched on as well. 

 &DIPOLE_TRANSITIONMOMENTS
    SHOW_VALENCE_EXCITATIONS      =         TRUE # (the default)
    SHOW_CORE_EXCITATIONS         =        FALSE # (the default)
    MAX_VALENCE_EXCITATION_ENERGY =     10.00000 # (the default)
    CORE_EXCITATION_FROM_E_MIN    =      0.00000 # (the default)
    CORE_EXCITATION_FROM_E_MAX    =      0.00000 # (the default)
    CORE_EXCITATION_TO_E_MIN      =      0.00000 # (the default)
    CORE_EXCITATION_TO_E_MAX      =      0.00000 # (the default)
    MAKE_XES_SPECTRA              =        FALSE # (the default)
    CORE_HOLE_IRREP               =            0 # (the default)
    CORE_HOLE_SPIN                =            0 # (the default)
    CORE_HOLE_PARTNER             =            0 # (the default)
    I_CORE_HOLE                   =            0 # (the default)
    X                             =      0.00000 # (the default)
    Y                             =      0.00000 # (the default)
    Z                             =      0.00000 # (the default)
 /DIPOLE_TRANSITIONMOMENTS

To select valence excitations from occupied to empty orbitals, set SHOW_VALENCE_EXCITATIONS. All Excitations up to MAX_VALENCE_EXCITATION_ENERGY will be printed. [eV]

To select core excitations, set SHOW_CORE_EXCITATIONS. All excitation from orbitals in the energy range between CORE_EXCITATION_FROM_E_MIN and CORE_EXCITATION_FROM_E_MAX to orbitals in the energy range between CORE_EXCITATION_TO_E_MIN and CORE_EXCITATION_TO_E_MAX are printed. [eV]

MAKE_XES_SPECTRA
Calculate a XES spectrum. The (imaginary) core hole is defined by the next 4 items.
CORE_HOLE_IRREP
Irrep of the core hole.
CORE_HOLE_SPIN
Spin of the core hole.
CORE_HOLE_PARTNER
Partner of the core hole.
I_CORE_HOLE
Number of the core hole within the irrep.
X,Y,Z
Unity vector, pointing in the direction of the Efield vector of the emitted light.

Main namelist for doing a fragment orbital analysis.

The fragment orbital analysis is only performed, if FRAG_ORB_ANALYSIS is set to true in the namelist PROPERTIES. The system under study must be devided into a arbitrary number of fragments by the user. Every atom of the complete system must be in exactly on fragment. The position of the atoms in the fragments must be the same as in the complete system. Of course all input parameters have to be the same in all inputs. Important: Set SAVE_EIGENVEC and SAVE_EIGENVEC_ALL to true (namelist RECOVER_OPTIONS) For every FRAGMENT one needs the file "saved_fragment.dat". GRID

This optional namelists starts the specification of the grid used in the SCF part. If no Post Hoc grid is specified, the SCF grid is used in the Post Hoc Part as well.

This namelist may only appear after the number of unique atoms was specified in namelists UNIQUE_ATOM_NUMBER. The grid must be specified together in one block. In case the optional namelist GRIDATOM appears, it must directly follow this namelist N_UNIQUE_ATOMS times. Namelist GRID may also be omitted and grid specification be started with namelist GRIDATOM.

Though there are default settings and grid specification is optional, it is highly recommended to do so. 

 &GRID
    SYM_REDUCE   = FALSE # (the default)
    WEIGHT_GRADS = FALSE # (the default)
 /GRID (namelist may be dropped)
SYM_REDUCE
If set to true, the local symmetry of a atom will be fully exploided to reduce the size of the integration grid. This may save a lot of CPU time. But be carefull: This option is not fully tested. Together with WEIGHT_GRADS=.true. no problems should occur, but if WEIGHT_GRADS is set to false, a rotation of the grid may take place and lead to a slight change of the total energy.
WEIGHT_GRADS
If set to true, the derivatives of weights of the integration grid will be considered in the calculation of the energy gradients with respect to nuclear displacements. The agreement between total energy and gradients will be increased by this option. As a price the CPU time consumption is rising. Attention: This switch affects the orientation and the construction of the POST_HOC and the SCF grid (no rotational invariannce, no BECKE scaling). If you are calculating binding energies, in all inputs WEIGHT_GRADS should be set uniformly to true (or false), also if you are not calculating GRADIENTS or the POST_HOC part. Note that you may set this switch here or in namelist GRID_PH but it will always affect both grid parts.


GRIDATOM

This optional namelists specifies the grid for one unique atom used in the SCF part. If no Post Hoc grid is specified, the SCF grid is used in the Post Hoc Part as well.

This optional namelist may only appear after the number of unique atoms was specified in namelists UNIQUE_ATOM_NUMBER and appear N_UNIQUE_ATOMS times in direct consecution. In case the optional namelist GRID appears, it must directly precede this namelist since the grid must be specified together in one block.

Though there are default settings and grid specification is optional, it is highly recommended to do so. 

 &GRIDATOM # unique atom   1
    NRAD    =         30 # (the default)
    NANG    =        131 # (the default)
    NPART   =          1 # (the default)
    RADIUS  = 0.00000000 # (the default)
    RSLATER =      TRUE # (the default)
    RPOPLE  =      FALSE # (the default)
    RIONIC  =      FALSE # (the default)
 /GRIDATOM
NRAD
NRAD determines the number of shells of the integration grid. A larger number leads to a better grid. The minimal value is 10. You are on the save side, if you choose 50 as a standard value and 70 for hydrogen and heavy elements (Au,...).
NANG
NANG determines the number of grid points per shell. Valid values are 31,51,71,91,111,131,151,171,191,231 and 291. A higher value leads to a better grid. A good choice is 171 for all atoms.
NPART
It is possible to use different NANG values in dependence from the distance to the nucleus. NPART is the number of domains and must be 1 or 5. As a result near the nucleus the grid will be rather sparse but in the outer regions very good.
RADIUS
Radius of the atom (in au), which will be used for the generation of the grid. Usually it has not to be specified.
RSLATER
If true (the default), atomic radii from Slater will be used for the generation of the grid.
RPOPLE
If true, atomic radii from POPLE will be used for the generation of the grid.
RIONIC
If true, iionic radii will be used for the generation of the grid. This can be used, if one wants to calculate very ionic systems. But be carefull: The NRAD parameter has to be increased to keep the total number of shells constant.

UNIQUE_ATOM_BASIS

This namelist is part of the orbital or charge or exchange fit function basis set specification of one unique atom. It describes the basis functions of one angular momentum or, in case of fitfunctions, also the r2- type basis.

The namelist may only appear within the basis set specification block started with namelist UNIQUE_ATOM_BASISSET. The syntax of this block is described there. 

 &UNIQUE_ATOM_BASIS
    N_EXPONENTS         =    5
    N_UNCONTRACTED_FCTS =    0 # (the default)
    N_CONTRACTED_FCTS   =    3
 /UNIQUE_ATOM_BASIS
  # Exponents
    2.000000000000000E-01    5.000000000000000E-01    1.250000000000000E+00  %
    3.125000000000000E+00    7.812500000000000E+00
  # Contraction   1
    1.000000000000000E+00    5.000000000000000E-01    2.000000000000000E-01  %
    1.000000000000000E-01    5.000000000000000E-02
  # Contraction   2
    1.000000000000000E-01    5.000000000000000E-01    1.000000000000000E+00  %
    5.000000000000000E-01    1.000000000000000E-01
  # Contraction   3
    5.000000000000000E-02    1.000000000000000E-01    2.000000000000000E-01  %
    5.000000000000000E-01    1.000000000000000E+00
N_EXPONENTS
Total number of exponents of primitive basis functions of the particular angular momentum and type (orbital, charge fit basis, exchange fit basis).
N_UNCONTRACTED_FCTS
Number of those exponents used without being contracted. The first N_UNCONTRACTED_FCTS exponents will be used.
N_CONTRACTED_FCTS
Number of contracted basis functions of the particular angular momentum and type (orbital, charge fit basis, exchange fit basis). In the case of charge and exchange fit functions so called "global contractions" exist as well. However, if global contractions are used for one unique atom (i.e. N_GLOB_CONS_CH / N_GLOB_CONS_XC is not zero) the use of the "local contractions" described here is forbidden (.i.e. N_CONTRACTED_FCTS must be zero).

The namelist must directly followed by:

  • one line containing N_EXPONENTS exponents for the gauss functions of the primitive basis.
  • N_CONTRACTED_FCTS lines containing N_EXPONENTS contraction coefficients. Each line describes how one contracted basis function is build as a linear combination of primitive basis functions with the given coefficients.
In the example above, continuation lines were used. The number format is quite free.

This namelist starts the specification of the orbital and fit function basis sets of one unique atom. The basis for orbitals and charge and exchange fitfunction are independent of each other but are all specified here. It is obligatory to specify the basis sets for all unique atoms.

The basis specification block may only appear after the number of unique atoms N_UNIQUE_ATOMS was specified in namelists UNIQUE_ATOM_NUMBER. N_UNIQUE_ATOMS blocks starting with namelist UNIQUE_ATOM_BASISSET must appear in direct consecution.

 &UNIQUE_ATOM_BASISSET
    LMAX_OB        =   2
    LMAX_CH        =   2
    LMAX_XC        =   2
    N_GLOB_CONS_CH =   0 # (the default)
    N_GLOB_CONS_XC =   0 # (the default)
 /UNIQUE_ATOM_BASISSET
LMAX_OB
The maximal angular momentum of orbital basis.
LMAX_CH
The maximal angular momentum of charge fit basis.
LMAX_XC
The maximal angular momentum of exchange fit basis.
N_GLOB_CONS_CH
The number of global contractions of charge fit basis.
N_GLOB_CONS_XC
The number of global contractions of exchange fit basis.

The basis set for each unique atom is specified in one block consisting of one namelist UNIQUE_ATOM_BASISSET followed in this order by:

  • (LMAX_OB + 1) namelists UNIQUE_ATOM_BASIS for the orbital basis for l=0,1,...
  • (LMAX_CH + 2) namelists UNIQUE_ATOM_BASIS for the charge fitfunction basis. The first namelist specifies the r2 basis and the following ones the basis for l=0,1,... .
  • (LMAX_XC + 2) namelists UNIQUE_ATOM_BASIS for the exchange fitfunction basis. The first namelist specifies the r2 basis and the following ones the basis for l=0,1,... . The exchange fit function basis is read in even if it is not used because the exchange potential is calculated numerically.
  • N_GLOB_CONS_CH namelists UNIQUE_ATOM_GLOB_CON namelists describing the global contractions of the charge fitfunctions.
  • N_GLOB_CONS_XC namelists UNIQUE_ATOM_GLOB_CON namelists describing the global contractions of the exchange fitfunctions.
Each of the namelists UNIQUE_ATOM_BASIS and UNIQUE_ATOM_GLOB_CON is directly followed by lines containing coefficients as described with these namelists.



UNIQUE_ATOM_GLOB_CON

This namelist is part of the charge or exchange fit function basis set specification of one unique atom. It describes one global contraction of either charge or exchange fitfunctions of one unique atom.

Global contractions are linear combinations of primitive basis functions of several different angular momenta. In the normal case you want linear combinations within one angular moment only, it is recommended to use local contractions described in namelist UNIQUE_ATOM_BASIS instead for the sake of better performance and clarity. Within one unique atom, it is not allowed to mix local and global contractions of either charge or exchange fitfunctions.

The namelist may only appear within the basis set specification block started with namelist UNIQUE_ATOM_BASISSET. The syntax of this block is described there. 

 &UNIQUE_ATOM_GLOB_CON
    N_CONTRIBUTING_FCTS = 2
 /UNIQUE_ATOM_GLOB_CON
 # l  if  exp  coef
   1  1   1    0.5E0
   1  1   2    0.5E0
N_CONTRIBUTING_FCTS
number of primitive basis functions contributing to the contraction

The namelist must directly be followed by N_CONTRIBUTING_FCTS lines each describing one contributing primitive basis function. Each line must contain the following in this order:

  • Index of angular momentum. Note that -1 means l=0, 0 means r2-function
  • Index of independent function of same IRREP, partner and angular momentum
  • Index of exponents contributing. This refers to exponents in corresponding namelist UNIQUE_ATOM_BASIS.
  • Coefficient to weight this primitive function with in the summation.



UNIQUE_ATOM_INDP_FCT

This outdated namelist is part of the description of the symmetry adaption coefficients that is only necessary in case no symmetry part is run (See namelist OPERATIONS).

It specifies the number N_FCTS of independent functions of the same IRREP, unique atom, angular momentum and exponent / contraction.

The symmetry adaption coefficients block may only appear after the number of unique atoms N_UNIQUE_ATOMS was specified in namelist UNIQUE_ATOM_NUMBER and the basis set (See namelists UNIQUE_ATOM_BASISSET), the symmetry group and Irreps (See namelist SYMMETRY_DATA) are known.

This namelist is read in in a 3-fold loop running is the following order over:

  • All N_UNIQUE_ATOMS unique atoms.
  • All N_IRREP Irreps.
  • All Max(LMAX_OB,LMAX_CH,LMAX_XC) angular momenta of the given unique atom.

Each namelist UNIQUE_ATOM_INDP_FCT must directly be followed by N_FCTS namelists UNIQUE_ATOM_SYMADAPT. All the symmetry adaption description must be written together in one block. 

 &UNIQUE_ATOM_INDP_FCT
    N_FCTS = 2
 /UNIQUE_ATOM_INDP_FCT

In case the integrals are not calculated (See namelist OPERATIONS), a section describing the renormalization coefficients for the orbital and fit function basis directly follows:

Lines of coefficients are read in in a 4-fold loop running is the following order over:

  • All N_UNIQUE_ATOMS unique atoms.
  • The type of basis: first orbitals, then charge fitfunctions and finally exchange fitfunctions.
  • The angular momenta up to LMAX_OB / LMAX_CH / LMAX_XC. In case of fitfunctions the order is as follows: l=0, r2, l=1, ...
  • All N_IRREP Irreps.

Within this loop, two lines are read in if the number of floats contained is greater zero:

  • The first line contains N_EXPONENTS coefficients in case global contractions are used (i.e. N_GLOB_CONS_?? greater zero) and N_UNCONTRACTED_FCTS coefficients otherwise. These are renormalization coefficients for uncontracted basis functions.
  • The second line contains N_CONTRACTED_FCTS coefficients for the (locally) contracted basis functions.

XC_CONTROL

Selects the exchange correlation functional used in the SCF part. If not specified otherwise (see namelist PHXC_CONTROL) the same potential is also used for the post hoc part.

Note the philosophy of the namelist: To permit a maximum flexibility in the choice of the functionals, there are no predefined combinations of exchange correlation functionals. The user is responsible for the selection of a meaningfull combination. For example, to perform a "VWN" calculation VNW and XALPHA must be set to true, as VWN only contains the correlation and does not include the LDA exchange. If you want to use a GGA correction, also the LDA functionals have to be selected. 

 &XC_CONTROL

    PWLDAc         = FALSE # (the default) /* version >= V20_delta */

    PERDEWWANG91C  = FALSE # (the default)
    PERDEWWANG91X  = FALSE # (the default)
    BAERENDS94     = FALSE # (the default)
    PBEx           = FALSE # (the default) /* version >= V20_delta */
    PBEc           = FALSE # (the default) /* version >= V20_delta */

Describing the Geometry

ParaGaus allows to use the known natural symmetry of the molecule in order to reduce the computational effort. In case of SCF convergence problems due to an over-complete basis set the use of symmetry may also help.

The symmetry is also used in the input geometry description of atoms and point charges.

The atoms or charges are grouped together as so-called unique atoms / charges that share the same name, nuclear charge and (for atoms) basis set and numerical grid. Moreover, all of the atoms or charges within this group must be symmetry-equivalent, i.e. they must be the complete set of centers that can be obtained from one center within this group using the symmetry transformations of the chosen symmetry group. The atoms / charges within one unique atom / charge group are called symmetry equivalent atoms / charges.

This is used in the input. The number of unique atoms / charges must be specified. Then for each unique atom / charge a namelist UNIQUE_ATOM / POINTCHARGE is used to specify the name, nuclear charge and number of equal charges and the coordinate of one equal atom / charge. The coordinates of the remaining equal atoms / charges are automatically calculated using the symmetry transformations of the chosen symmetry group. The specified number of equals is used for a cross-check only. For each unique atom, an own basis set and numerical grid must also be given.

The set of symmetry transformations within one symmetry group are not completely defined by the symmetry group alone. The location and orientation of the rotation axis, planes and centers of symmetry is still open. Therefore conventions are introduced for each symmetry group:
